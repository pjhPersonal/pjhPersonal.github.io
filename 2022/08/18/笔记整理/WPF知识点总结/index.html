WPF知识点总结

1. 绑定（Binding）

（1）在WinForm中，我们要想对控件赋值，需要在后台代码中拿到控件对象进行操作，这种赋值形式，从根本上是无法实现界面与逻辑分离的。在WPF中，微软引入了Binding对象，通过Binding，我们可以直接将控件与数据做绑定，在后台不用拿到控件对象，直接操作数据源，系统会自动监听数据源的变化实时的更新到控件上，相反的，用户改变控件的值，也会实时更新到后台数据源，这也就是我们所说的，WPF天生支持MVVM模式。

（2）使用Binding的一般流程：

  1. 实现INotifyPropertyChanged接口。
  2. 为需要绑定的属性的set块添加PropertyChanged事件通知。
  3. 绑定数据源（XAML或者backend）。
  • ObservableCollection是实现INotifyCollectionChanged接口的数据集合的内置实现，表示一个动态数据集合。在添加项、移除项或刷新整个列表时，此集合将提供通知。

2. 依赖属性（Dependency Property）

（1）属性：

  1. 属性用于封装字段，通过属性来验证数据的合法性或控制对外的访问性等等。
  2. 每个属性的背后都有其对应的一个字段做支撑。就算是自动属性，编译时系统也会创建其字段。
  3. 在C#中，属性最后会编译成两个方法：get_{属性名}和set_{属性名}（如果是只读属性，则没有set方法）。

（2）依赖属性：

  1. 每一个控件都有很多属性，有属性就有字段的内存开销。但实际上对于一个控件，大多数只会使用其部分常用的属性，那些不经常使用的属性相当于白白占用着内存。
  2. 对此，微软在WPF中引入了依赖属性，依赖属性允许没有自己的字段，可以通过Binding绑定到其他对象的属性或者说数据源上，从而获得值。这种依赖在其他对象上的属性，就是依赖属性。依赖属性可以没有自己的字段，在使用时可以通过Binding从别的对象身上获取，给自己临时创建内存空间，这样不使用就不会有多余的内存消耗。
  3. 如果直接给依赖属性赋值，它依然如属性一样，会占用空间。必须使用Binding才能实现“依赖”。

3. 逻辑树和视觉树

（1）逻辑树：

  1. Window - StackPanel - Label - String
  2. 逻辑树可以认为是XAML所见的。
  3. 逻辑树始终存在于WPF的UI中，不管UI是用XAML编写还是用代码编写。

（2）视觉树：

  1. Window - Border - AdornerDecorator - ContentPrensenter - StackPanel - …
  2. 视觉树基本上是逻辑树的一种扩展。逻辑树的每个节点都被分解为它的核心视觉组件，暴露了视觉的实现细节。
  3. 并不是所有的逻辑树节点都可以扩展为视觉树节点。只有从Visual和Visual3D继承的元素才能被视觉树包含。其他元素不能包含是因为它们本身没有自己的Rendering行为。
    • e.g. 任何继承自ContentElement的东西都可以在UI上显示，但并不在视觉树中，因为ContentElement不是继承自Visual或Visual3D。为了显示在UI上，ContentElement必须寄宿在一个Visual主体上，通常是一个FrameworkElement。因此对于ContentElement，无法使用VisualTreeHelper来遍历视觉树，只能沿着逻辑树查找ContentElement的父元素，直到找到一个Visual对象才能继续使用VisualTreeHelper。

（3）遍历：

  1. 逻辑树的遍历使用LogicalTreeHelper。
  2. 视觉树的遍历使用VisualTreeHelper。

4. 路由事件（Routed Event）

（1）路由事件是一种可以针对元素树中的多个侦听器（而不是仅针对引发该事件的对象）调用处理程序的事件。

（2）路由事件沿着视觉树传递。

（3）路由策略：

  1. 冒泡（Bubbling）：路由事件由事件激发者出发向它的上一层容器一层一层路由，直至最外层的容器（Windows或Page）。因为是由树的底部想树的顶部移动，而且从事件激发元素到UI树的树根只有确定的一条路径。
    • e.g. MouseDown
  2. 直接（Direct）：模仿CLR直接事件，直接将事件消息送达事件处理器。
    • e.g. trigger, VisualState
  3. 隧道（Tunneling）：事件的路由刚好和冒泡式相反，是由树的树根向事件激发者移动，这就想当于在树根和目标控件之间挖了一条隧道，事件只能沿着隧道移动。
    • e.g. PreviewMouseDown
  • 冒泡事件和隧道事件一般成对出现。同一事件，执行时首先是隧道事件，然后是冒泡事件。

5. 多线程相关

在WPF应用程序中，主线程负责接受输入、处理事件、绘制屏幕等工作。为了使主线程及时相应，防止假死，在开发过程中对一些耗时、消耗资源比较多的操作，都会去创建一个或多个子线程去完成操作。由于UI界面是主线程创建的，所以子线程不能直接更新由主线程维护的UI界面。

（1）Dispatcher：

  1. Dispatcher的作用是用于管理线程工作项队列。
  2. 在WPF中，所有的WPF对象都派生自DispatcherObject，DispatcherObject暴露了Dispatcher属性用来取得创建对象线程对应的Dispatcher。DispatcherObject对象只能被创建它的线程所访问，其他线程修改 DispatcherObject需要取得对应的Dispatcher，调用Invoke或者BeginInvoke来投入任务。
  3. 为了能在一个线程更新另一个线程创建的对象的信息，Dispatcher提供了两个方法：Invoke和BeginInvoke。Invoke是同步操作，而BeginInvoke是异步操作。这两个操作按指定的priority添加到Dispatcher的队列中。
